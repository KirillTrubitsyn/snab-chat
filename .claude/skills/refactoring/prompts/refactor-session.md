# Промпт для AI-ассистированной сессии рефакторинга

Используй этот промпт при выполнении рефакторинга с Claude Code / Cursor / Windsurf / аналогами. Структура соответствует Explore-Plan-Execute workflow: сначала план, только потом изменения.

## Шаблон

```
ROLE
Ты выступаешь в роли senior engineer, выполняющего дисциплинированный рефакторинг. Цель — изменить структуру кода, не меняя поведение. Любое изменение семантики — ошибка.

CONTEXT
Проект: <коротко: стек, назначение>
Корневая папка: <путь>
Затрагиваемая область: <например: src/features/billing/>
Запрещено трогать: <пути, которые не должны меняться в рамках этой сессии>

GOAL
<одним предложением: что именно рефакторим>
Пример: «Выделить логику вычисления скидок из `src/billing/discount.ts:applyDiscount` в три чистые функции, сохранив сигнатуру и поведение публичной функции `applyDiscount`.»

METHODOLOGY
<какая методология применяется>
Пример: Extract Function + Split Phase. См. references/methodologies.md.

CONSTRAINTS
- Поведение должно быть идентичным. Все существующие тесты должны проходить до и после каждого коммита.
- Сигнатура и тип возврата публичных функций не меняются в этой сессии.
- Не удаляй код, не добавляй новые зависимости, не меняй конфиги.
- Строго запрещено: «заодно улучшить», «попутно переименовать», «добавить обработку ошибки».
- Архитектурные решения (DI, новые слои, замена библиотек) — не в рамках этой сессии.

SAFETY NET STATUS
- Тесты: <какие, где, как запустить>
- Coverage целевой области: X%
- tsc --noEmit: зелёный / ошибки в <файле>
- Линтинг: <состояние>

WORKFLOW
1. PLAN (read-only). Прочитай целевые файлы. Составь план рефакторинга как нумерованный список микро-шагов. Каждый шаг:
   — Один атомарный рефакторинг (из references/refactoring-catalog.md).
   — После его применения код компилируется и тесты проходят.
   — Отдельный коммит.
   Опиши план в формате:
     Шаг N: <название рефакторинга>
     Файл(ы): <пути>
     Что меняется: <краткое описание>
     Как проверяем: <команда тестов / tsc>
     Коммит-сообщение: <сообщение>
   Не вноси никаких изменений, пока план не одобрен.

2. AWAIT APPROVAL. Остановись после плана. Жди моего явного «approved» или правок.

3. EXECUTE. Выполни план строго по шагам. После каждого шага:
   — Запусти: <конкретная команда тестов + tsc>
   — Если зелёное: коммит с заявленным сообщением.
   — Если красное: откатись (git reset --hard HEAD) и сообщи об этом. Не пытайся чинить внутри сессии.

4. REPORT. По завершении: список выполненных шагов, результаты тестов до/после, diff-статистика по файлам.

ANTI-DRIFT RULES
- Не трогай файлы за пределами указанных.
- Не «почини по пути» замеченные проблемы. Запиши их в конце отчёта как follow-up.
- Никаких новых импортов, кроме тех, что требуются для объявленного рефакторинга.
- Никаких format-изменений (prettier) внутри refactor-коммитов.
- Если в процессе понимаешь, что план неполный — остановись и верни управление, не импровизируй.

START
Начни с шага 1 (PLAN).
```

## Рекомендации по применению

1. **Один рефакторинг на сессию**. Не смешивай «вытащим хук» и «переименуем тип». Второй рефакторинг — отдельный промпт.

2. **Узкие границы файлов**. Явно перечисли папки, которые можно трогать. AI склонен «заодно» обновлять соседние файлы.

3. **Plan Mode обязателен**. В Claude Code активируй Plan Mode (Shift+Tab до появления режима). В Cursor — используй feature «Plan». План, прочитанный человеком, убирает 80% проблем.

4. **Жёсткий требование идентичного поведения**. В каждый промпт добавляй фразу: `behavior must be identical; run the full test suite before and after each step; if any test fails, revert and stop`. Это критичный анти-drift.

5. **Маленькие сессии**. Крупный рефакторинг — это цепочка маленьких сессий, не одна длинная. Чем дольше сессия, тем выше риск контекстного дрейфа.

6. **Отдельные сессии для reviewer'а**. После того как сессия завершена, открой новую сессию с другим промптом: «Ты reviewer. Вот diff, вот требования. Проверь, что (а) поведение не изменилось, (б) запрещённых изменений нет, (в) план соблюдён». Разделение исполнителя и ревьюера снижает bias.

7. **Тесты раньше, не позже**. Если план содержит «в конце прогоним тесты» — это антипаттерн. Тесты должны идти после каждого шага.

8. **Feature flags для рискованных рефакторингов**. Если рефакторинг затрагивает runtime-поведение (миграция стора, замена библиотеки) — оборачивай в feature flag, чтобы переключить на новую реализацию отдельным решением.

## Что не делегировать AI

Есть категории решений, которые принимает человек, даже если AI выполняет механику:

- Выбор стратегии (strangler fig vs branch by abstraction vs big bang).
- Выбор архитектурных границ (что стало бы отдельным модулем, где проходит API).
- Security-критичный код (auth, crypto, permissions).
- Миграции БД, которые не откатываются.
- Публичные API, которые используются внешними клиентами.
- Код, не покрытый тестами и без возможности написать характеризационные быстро.
- Hotspots с неясной историей и без владельцев в команде.

Во всех этих случаях AI может помогать с мелкими механическими шагами внутри решения, принятого человеком, но не принимать решение сам.

## Пример завершённой сессии

Промпт (заполненный):

```
ROLE
Ты выступаешь в роли senior engineer, выполняющего дисциплинированный рефакторинг...

CONTEXT
Проект: internal SaaS на Next.js 15 + Drizzle + PostgreSQL
Корневая папка: /Users/me/projects/saas
Затрагиваемая область: src/features/billing/
Запрещено трогать: src/app/, src/components/ui/, drizzle/

GOAL
Выделить логику вычисления скидок из src/features/billing/discount.ts:applyDiscount
в три чистые функции: validateDiscountInput, calculateDiscountedTotal, persistDiscount.
Сигнатура applyDiscount и её поведение не меняются.

METHODOLOGY
Extract Function + Split Phase.

CONSTRAINTS
- Поведение идентично. Все тесты pnpm test должны проходить после каждого шага.
- ...

SAFETY NET STATUS
- Тесты: pnpm test (Vitest + Playwright)
- Coverage src/features/billing/: 82%
- tsc --noEmit: зелёный
- Характеризационный тест на applyDiscount был добавлен вчера (файл tests/billing/applyDiscount.char.test.ts, 25 кейсов)

WORKFLOW
...

START
```

Ожидаемый ответ AI:

```
PLAN:

Шаг 1: Extract Function — validateDiscountInput
Файлы: src/features/billing/discount.ts
Что меняется: выделяю строки 45-67 (валидация ввода) в новую функцию validateDiscountInput(input: DiscountInput): void.
Как проверяем: pnpm test -- billing && pnpm tsc --noEmit
Коммит-сообщение: refactor(billing): extract validateDiscountInput

Шаг 2: Extract Function — calculateDiscountedTotal
Файлы: src/features/billing/discount.ts
Что меняется: выделяю строки 69-110 (расчёт) в calculateDiscountedTotal(order: Order, discount: Discount): number.
Как проверяем: pnpm test -- billing && pnpm tsc --noEmit
Коммит-сообщение: refactor(billing): extract calculateDiscountedTotal

Шаг 3: Extract Function — persistDiscount
Файлы: src/features/billing/discount.ts
Что меняется: выделяю строки 112-160 (запись + уведомление) в persistDiscount(...): Promise<void>.
Как проверяем: pnpm test -- billing && pnpm tsc --noEmit
Коммит-сообщение: refactor(billing): extract persistDiscount

Шаг 4: Simplify applyDiscount
Файлы: src/features/billing/discount.ts
Что меняется: тело applyDiscount сводится к вызовам трёх новых функций последовательно.
Как проверяем: pnpm test && pnpm tsc --noEmit
Коммит-сообщение: refactor(billing): simplify applyDiscount to three-phase composition

Waiting for approval.
```

Ты: «approved».

AI выполняет шаги, отчитывается.
