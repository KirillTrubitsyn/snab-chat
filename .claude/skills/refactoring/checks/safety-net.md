# Safety Net — оценка перед рефакторингом

## Цель

Рефакторинг без safety net — это лотерея. Прежде чем менять структуру кода, нужно убедиться, что регрессию заметят. Этот модуль помогает оценить, достаточен ли существующий safety net, и что делать, если нет.

## Что проверять

### 1. Автоматические тесты

- Какие типы тестов есть: unit, integration, e2e, contract, snapshot?
- Прогоняются ли они локально без ошибок прямо сейчас? Зафиксируй baseline.
- Какое покрытие у затрагиваемого модуля? Запусти coverage-прогон:
  ```
  # Vitest/Jest
  npx vitest run --coverage
  npx jest --coverage

  # Python
  pytest --cov=<module> --cov-report=term-missing
  ```
- Покрытие **строк** < 70% по затрагиваемой области — рефакторинг рискован.
- Покрытие **ветвлений** < 60% — рефакторинг условий особенно рискован.

### 2. Характер тестов

Покрытие в процентах само по себе ничего не значит. Проверь:

- Тесты проверяют **поведение** или **реализацию**? Тесты на реализацию (mock-heavy, проверка вызовов приватных методов) ломаются при рефакторинге, хотя поведение не изменилось. Это антипаттерн.
- Есть ли тесты, которые пропущены / заскипаны (`.skip`, `xit`, `@pytest.mark.skip`)?
- Запускаются ли тесты в CI на каждый PR?

### 3. Типизация

Для TS:

- Проходит ли `pnpm tsc --noEmit` без ошибок?
- Какой уровень strict-настроек? Включены ли: `strict`, `noUncheckedIndexedAccess`, `exactOptionalPropertyTypes`, `noImplicitOverride`?
- Сколько `any` в кодовой базе? `grep -rn ": any" --include="*.ts" --include="*.tsx" | wc -l`
- Сколько `as` type assertions? `grep -rn " as [A-Z]" --include="*.ts" --include="*.tsx" | wc -l`

Для Python:

- Есть ли mypy / pyright? Проходит ли `mypy --strict`?

### 4. Линтинг

- Biome / ESLint / Ruff настроены? Fail-on-warning?
- Какие правила отключены? Часто отключают именно те правила, которые ловят проблемные конструкции.

### 5. Mutation testing (бонус)

Если настроен Stryker (JS/TS) или mutmut (Python) — это золотой стандарт оценки качества тестов. Mutation score < 70% = тесты пропускают много регрессий.

## Что делать, если safety net слабый

### Сценарий A: legacy-код без тестов

Используй **characterization tests** (Michael Feathers, Emily Bache):

1. Возьми целевую функцию / модуль.
2. Сгенерируй репрезентативный набор входов (реальные данные из production-логов, граничные случаи, случайные генерации через property-based).
3. Запусти код, зафиксируй выходы.
4. Сохрани пары (вход, выход) как golden master / approval-файлы.
5. Тест = пропустить вход через код и сравнить с golden.

Библиотеки:

- **JS/TS**: `jest-image-snapshot`, `ava snapshot`, `vitest toMatchSnapshot`, `approvaltests-js`
- **Python**: `pytest --snapshot`, `approvaltests`
- Для любого языка: просто сохраняй JSON/text fixtures и сравнивай.

Characterization tests фиксируют текущее поведение «как есть» — даже с багами. Это нормально: цель — не изменить поведение при рефакторинге. Если баги известны — их исправляют отдельным коммитом после рефакторинга.

### Сценарий B: есть тесты, но mock-heavy

Тесты, которые проверяют «был ли вызван метод X с аргументами Y», ломаются при любом рефакторинге. Перед рефакторингом:

1. Замени mock-тесты на тесты через публичный API модуля.
2. Используй integration-тесты с реальными зависимостями где возможно (in-memory БД, testcontainers).
3. Mock оставь только для внешних систем (HTTP, почта, платежи).

### Сценарий C: нет типов

Прежде чем рефакторить нетипизированный код:

1. Включи TS / mypy локально для модуля.
2. Добавь типы на публичные границы функций.
3. Это само по себе вскроет часть багов и даст намного больше уверенности.

### Сценарий D: есть CI, но проверки слабые

Добавь gates в pipeline:

- `tsc --noEmit` — блокирующий.
- `biome check` / `eslint --max-warnings 0` — блокирующий.
- Тесты — блокирующие.
- Coverage — не регрессирует (сравнение с main).
- Bundle size — не регрессирует неожиданно.

## Контрольный список перед началом рефакторинга

```
[ ] Тесты проходят локально (baseline зафиксирован)
[ ] Тесты проходят в CI
[ ] Покрытие затрагиваемой области ≥ 70% по строкам
[ ] tsc --noEmit — зелёный (если TS)
[ ] Линтер — зелёный
[ ] Есть точка отката (git checkpoint, stash-план)
[ ] Рефакторинг идёт в отдельной ветке
[ ] Согласована цель рефакторинга (не «улучшу как получится»)
```

Если хотя бы два пункта не выполнены — сначала построй safety net, только потом рефактори.
