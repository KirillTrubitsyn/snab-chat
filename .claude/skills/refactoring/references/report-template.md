## 6. Детальный разбор

### 6.1 `<название находки>`

**Файл**: `<путь>:<строки>`
**Категория**: Long method / Prop drilling / ...
**Severity**: High
**Effort**: M
**Risk**: Low

**Описание**. Что именно представляет собой smell и почему это проблема для команды / продукта.

**Doказательство**. Цитата кода:

```ts
// src/billing/discount.ts:45-180
function applyDiscount(...) {
  // 135 строк смешанной логики:
  // - валидация
  // - расчёт
  // - запись в БД
  // - отправка уведомления
  // - логирование
}
```

**Методология**: Extract Function + Split Phase.

**План рефакторинга**:

1. Выделить `validateDiscountInput(...)` — чистая функция.
2. Выделить `calculateDiscountedTotal(...)` — чистая функция.
3. Вынести запись и уведомление в `persistDiscount(...)`.
4. Основная функция `applyDiscount` становится композицией из трёх шагов.

**Проверка сохранения поведения**:

- Перед рефакторингом: характеризационный тест на `applyDiscount` с 20 репрезентативными входами.
- После каждого шага: прогон всего testsuite, включая e2e-сценарии оформления заказа.
- Snapshot БД: таблица `discounts` должна содержать те же записи при тех же входах.

**Коммиты (ожидаемая цепочка)**:

```
refactor(billing): add characterization tests for applyDiscount
refactor(billing): extract validateDiscountInput
refactor(billing): extract calculateDiscountedTotal
refactor(billing): extract persistDiscount
refactor(billing): simplify applyDiscount to three-phase composition
```

## 7. Рекомендуемый порядок работ

Очередь в порядке убывания ROI:

1. **Первая волна** (не блокирует фичи, даёт быстрый выигрыш):
   - Rename + Extract Variable в hotspot-файлах — 2 дня.
   - Устранение dead code через `knip` — 1 день.
   - Замена любых `any` в публичных API на unknown + narrowing — 3 дня.

2. **Вторая волна** (требует планирования со стороны продукта):
   - Выделение `useUser` hook и удаление prop drilling — 1 спринт.
   - Миграция server state на TanStack Query — 1 спринт.

3. **Третья волна** (архитектурные изменения, требуют решения):
   - Strangler fig для модуля `legacy-payments` → `payments-v2` — 2 спринта.
   - Внедрение feature-sliced структуры — 2 спринта.

Для каждой волны — отдельный план с milestone и критерием приёмки.

## 8. Технический долг, который не стоит трогать

Эти находки формально являются smells, но рефакторинг дороже выгоды или слишком рискован:

- `<область>`: редко меняется, работает стабильно, нет тестов и нет бизнес-драйвера. Рекомендация: оставить, не ломать.
- `<область>`: запланировано выведение из эксплуатации в Q3 — смысла в рефакторинге нет.

Явная фиксация этих решений важна, чтобы не возвращаться к ним снова.

## 9. Подготовительные работы (если safety net недостаточен)

## Рекомендации по использованию шаблона

- Короче лучше. Отчёт, который никто не дочитал, бесполезен. Подробности — в секциях, executive summary — на 1 страницу максимум.
- Каждая находка поддержана ссылкой на файл и строки. Никаких «где-то в платежах».
- Severity — по влиянию на команду / продукт, не по субъективному «мне не нравится».
- Отчёт должен быть runnable: рекомендуемые команды копируются и исполняются.
- Храни отчёты в репозитории в `docs/refactoring/` для исторической динамики.
