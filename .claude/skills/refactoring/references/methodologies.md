# Методологии рефакторинга

Этот файл описывает основные методологии. Выбирай под задачу. В одном крупном проекте обычно сочетаются несколько.

## 1. Red-Green-Refactor (Kent Beck)

Базовый цикл TDD, применимый к микро-рефакторингу.

```
1. Red    — напиши падающий тест, описывающий желаемое поведение
2. Green  — напиши минимальный код, чтобы тест прошёл
3. Refactor — улучши структуру, сохраняя зелёные тесты
```

**Когда применять**: при разработке новой функциональности и при мелких локальных рефакторингах, когда safety net уже есть.

**Чего не решает**: не подходит для legacy-кода без тестов, для крупной замены модуля, для миграции между архитектурами.

**Правила дисциплины**:

- В фазе Refactor не добавляй новое поведение.
- Если во время Refactor тест покраснел — откатись немедленно, не чини «по ходу».
- Коммить после каждой зелёной фазы. Микрокоммиты — норма.

## 2. Parallel Change (expand / migrate / contract)

Martin Fowler. Безопасное изменение сигнатуры публичного API, когда разом переключить всех потребителей невозможно.

```
1. Expand   — добавь новую форму рядом со старой (не удаляя старую)
2. Migrate  — переведи потребителей на новую форму по одному
3. Contract — удали старую форму, когда никто её не использует
```

**Пример**. Переименование параметра функции с `userId: string` на `user: User`:

```ts
// Было
function charge(userId: string, amount: number) { ... }

// Фаза Expand — поддерживаем оба входа
function charge(userOrId: string | User, amount: number) {
  const user = typeof userOrId === "string" ? findUser(userOrId) : userOrId;
  ...
}

// Фаза Migrate — обновляем вызывающих по одному, каждый вызов отдельным коммитом.

// Фаза Contract — когда все переехали, удаляем старую ветку
function charge(user: User, amount: number) { ... }
```

**Когда применять**: изменение сигнатуры функции, переименование столбца в БД, изменение формы API-ответа.

**Ключевой плюс**: каждый промежуточный коммит production-ready. Можно мержить в main в любой момент.

## 3. Branch by Abstraction (Dave Farley, Jez Humble)

Крупная замена реализации в активно разрабатываемом коде без feature-бранчей и без big-bang.

```
1. Введи абстракцию (интерфейс / функцию-прослойку) над старой реализацией
2. Перенаправь всех потребителей через абстракцию
3. Добавь новую реализацию рядом со старой
4. Переключи абстракцию с старой на новую (обычно через feature flag)
5. Удали старую реализацию
6. При желании — inline-и абстракцию, если она больше не нужна
```

**Пример**. Миграция с одной БД на другую, со старой auth-библиотеки на новую, с Redux на Zustand.

```ts
// Шаг 1. Абстракция
interface AuthProvider {
  signIn(email: string, password: string): Promise<Session>;
  signOut(): Promise<void>;
  getSession(): Promise<Session | null>;
}

// Шаг 2. Все потребители зовут AuthProvider, не библиотеку напрямую.

// Шаг 3. Две реализации.
class LegacyAuthProvider implements AuthProvider { ... }
class NewAuthProvider implements AuthProvider { ... }

// Шаг 4. Выбор в runtime, чаще через feature flag.
const authProvider: AuthProvider =
  flags.useNewAuth ? new NewAuthProvider() : new LegacyAuthProvider();
```

**Когда применять**: замена крупного компонента (БД, очередь, фреймворк, библиотека), миграция между двумя существующими состояниями.

**Почему работает**: код всё время production-ready. Переключение — один булев флаг. Откат мгновенный.

## 4. Strangler Fig (Martin Fowler)

Аналогия: фикус-душитель обвивает дерево, постепенно его замещает, в итоге стоит на его месте. Так мы постепенно замещаем legacy-систему новой, по одному фрагменту за раз.

```
1. Проведи «сейдж»: выбери первый ломтик функциональности для миграции (обычно малый и с чёткими границами)
2. Построй новую реализацию этого ломтика рядом со старой
3. Поставь между клиентом и системой «фасад», который маршрутизирует трафик: новые вызовы — в новую реализацию, остальные — в старую
4. Повторяй, перемещая по одному ломтику
5. Когда все ломтики переехали — отключи старую систему
```

**Когда применять**:

- Миграция монолита на модульный монолит или сервисы.
- Замена движка / фреймворка приложения (Rails → Node, Angular → React, Express → Fastify).
- Вывод legacy-модуля из эксплуатации.
- Любая ситуация, где большой кусок должен быть заменён, а остановить разработку нельзя.

**Техники реализации фасада**:

- HTTP reverse proxy (nginx, Cloudflare, Traefik) с правилами по пути.
- API gateway с постепенным перенаправлением.
- На уровне кода: адаптер-функция с условиями.
- В Next.js: одни роуты в `app/` (новые), другие — в `pages/` (старые); `next.config` помогает с rewrites.

**Правила**:

- Двойная запись (dual writes) при необходимости: новый код пишет и в старое, и в новое хранилище, пока переключение не завершено.
- Мониторинг расхождения между старой и новой реализациями на том же трафике (shadow traffic) — до переключения.
- Явный план: что мигрирует, в каком порядке, какой критерий успеха у каждого ломтика.

## 5. Mikado Method (Ola Ellnestam, Daniel Brolund)

Алгоритм для рефакторинга запутанного кода, где каждое изменение «тянет» за собой другое.

```
1. Сформулируй цель (Mikado goal): конечное состояние.
2. Попробуй сделать изменение напрямую — чаще всего сломается.
3. Запиши препятствия в дерево (prerequisites). Каждое препятствие — узел.
4. Откати попытку (git reset --hard).
5. Возьми лист дерева (узел без детей), сделай его — работает.
6. Откати, если тоже не получилось, и раскрой этот узел в поддерево.
7. Постепенно дерево строится снизу вверх. Реальные изменения идут с листьев.
```

**Ключевая идея**: делать только то, что **работает сразу**. Всё, что тянет зависимости — записать и отложить. Реальные коммиты — только работающие, зелёные, по одному шагу.

**Пример узла дерева**:

```
Goal: Выделить auth-логику из app/routes/*.ts в модуль app/auth/
├── Prerequisite: перенести middleware-функции
│   ├── Prerequisite: убрать из middleware прямые обращения к глобальному store
│   │   └── Prerequisite: извлечь store-зависимости в аргумент
│   └── Prerequisite: привести сигнатуру middleware к единому виду
└── Prerequisite: согласовать форматы session-cookie
    └── Prerequisite: миграция тестов, полагающихся на старый cookie-формат
```

Работают с листьями вверх.

**Когда применять**: legacy-код, где непонятно, за какую ниточку потянуть; архитектурные изменения в запутанной системе.

**Дисциплина**: никаких «длинных веток Mikado»; каждое изменение — отдельный коммит в main, если лист зелёный.

## 6. Characterization via Golden Master (Michael Feathers)

Не методология изменения, а методология safety net для legacy.

```
1. Подай на вход системе большой набор реалистичных вводов
2. Зафиксируй полный вывод как "золотой эталон"
3. Пиши тесты, которые сравнивают вывод с эталоном
4. Теперь рефактори, проверяя, что эталон не изменился
```

**Когда применять**: перед рефакторингом legacy, где нет тестов и не очевидно, как их писать.

**Инструменты**:

- approvaltests-js / ApprovalTests.Python / ApprovalTests.Java — от Emily Bache.
- Самописный runner на сериализации JSON/текста и сравнении.

## 7. Codemods (механизированные изменения)

Когда одно и то же изменение нужно применить в десятках и сотнях файлов, ручной рефакторинг — ошибка.

**Инструменты**:

- **ast-grep** — декларативные правила `pattern → rewrite`, быстрый, удобный для 80% случаев. CLI и YAML-правила.
- **ts-morph** — императивная работа с TypeScript AST, для сложных трансформаций.
- **jscodeshift** — классика для JS, богатая экосистема готовых трансформов (miграции React, Next.js).
- **comby** — структурный поиск-замена с поддержкой множества языков.
- **OpenRewrite** — Java/Kotlin/XML/YAML, recipe-ориентированный.

**Workflow**:

1. Найди один-два репрезентативных примера изменения.
2. Запиши трансформацию как правило (ast-grep) или скрипт (ts-morph / jscodeshift).
3. Прогони сначала в dry-run.
4. Прогон на подветке, полный прогон тестов.
5. Коммит как отдельный «mechanical refactor», отмечай в PR-описании, что изменение автоматическое.

**Когда применять**: миграции API (React 17→18, Next.js 13→14→15), переименования типов/функций по всей кодовой базе, устранение устаревших паттернов.

**Важно**: codemods не заменяют осмысленный рефакторинг. Механическое изменение — только то, что действительно механическое. Всё, что требует решения — руками.

## 8. AI-ассистированный рефакторинг

Это не самостоятельная методология, а техника исполнения. Обязательно сочетается с одной из выше. См. промпты в `prompts/`.

Ключевые правила:

- AI не заменяет выбор методологии. Выбираешь ты.
- AI хорош в: extract function, rename, inline, парная работа над codemod, генерация characterization tests.
- AI плох в: стратегические решения, архитектурные границы, выбор что рефакторить, критичная security-логика.
- Каждую AI-порцию — review руками, прогон тестов, явный коммит.

## Как выбирать методологию

| Ситуация | Методология |
|---|---|
| Локальное улучшение с хорошими тестами | Red-Green-Refactor |
| Изменение публичной сигнатуры | Parallel Change |
| Крупная замена при активной разработке | Branch by Abstraction |
| Миграция модуля / системы | Strangler Fig |
| Запутанный legacy, непонятно с чего начать | Mikado Method |
| Legacy без тестов | Characterization Tests первым слоем, затем одна из выше |
| Механическое изменение по N файлам | Codemods (ast-grep / ts-morph / jscodeshift) |
