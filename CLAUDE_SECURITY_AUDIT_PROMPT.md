# Промпт для Claude Code: полный аудит безопасности приложения

Скопируй текст ниже целиком и вставь в Claude Code.

---

Ты — senior application security engineer + red team reviewer.

Твоя задача: провести **комплексный аудит безопасности** моего приложения в текущем репозитории и выдать практический план устранения уязвимостей перед production.

## 1) Обязательный scope аудита

Проверь минимум следующие направления:

1. **Аутентификация и авторизация**
   - Можно ли обойти вход/роль через прямые API вызовы (мимо UI)
   - Broken Access Control, IDOR, horizontal/vertical privilege escalation
   - Сессии, токены, refresh-flow, logout invalidation, 2FA

2. **API и backend поверхность**
   - Открытые эндпоинты без защиты
   - Наличие rate-limit, anti-bruteforce, anti-enumeration
   - Валидация входа (schema validation), mass assignment
   - Ошибки: не утекают ли stack traces, SQL/internal messages

3. **Секреты и чувствительные данные**
   - Хардкоженные API keys, токены, пароли, private keys
   - Утечки в git-истории, документации, логах
   - Небезопасные `.env`, примеры конфигов, CI secrets

4. **Файловые загрузки и storage**
   - MIME spoofing, path traversal, unsafe file names
   - Signed URL misuse, public bucket issues, excessive permissions

5. **LLM / prompt-injection (если есть AI-функции)**
   - Prompt injection в документах/пользовательском вводе
   - XML/HTML/context breakouts
   - Data exfiltration через LLM-ответы

6. **Инфраструктурные и эксплуатационные риски**
   - Security headers (CSP, HSTS, X-Frame-Options, etc.)
   - CORS misconfig, Origin/Referer checks
   - WAF, DAST, key rotation, CI security gates

## 2) Что нужно сделать технически (обязательно)

- Проведи статический аудит кода и конфигов.
- Найди и перечисли все API route handlers и classify:
  - public
  - auth-protected
  - admin-only
- Для каждого потенциально опасного маршрута укажи:
  - точный файл и строки
  - вектор атаки
  - реальный impact
  - вероятность эксплуатации
- Отдельно проверь, можно ли пользоваться приложением/критичными методами **не через интерфейс**, а напрямую (curl/Postman/script).
- Отдельно проверь наличие секретов:
  - в рабочем дереве
  - в git-истории
  - в документации

## 3) Формат результата (строго)

Верни результат в Markdown со следующей структурой:

### A. Executive Summary (1 экран)
- Текущий уровень риска: Critical/High/Medium/Low
- Кол-во находок по уровням
- Что блокирует production прямо сейчас

### B. Таблица уязвимостей
Колонки:
- ID
- Severity
- CWE/OWASP
- Компонент
- Кратко
- Exploit scenario
- Impact
- Рекомендация
- Effort (S/M/L)

### C. Подробный разбор каждой уязвимости
Для каждой находки:
- Где (файл + строки)
- Почему это уязвимость
- Как эксплуатируется (по шагам)
- Что исправить (конкретно, без воды)
- Пример безопасного патча (если возможно)

### D. Проверка секретов
- Что именно искал
- Что найдено
- Что похоже на false positive
- Какие ключи нужно ротировать немедленно

### E. Проверка открытых эндпоинтов
- Список endpoint'ов без защиты
- Какие из них допустимо оставить публичными
- Какие срочно закрыть

### F. Production Hardening Plan
С приоритетами:
- **P0 (до релиза)**
- **P1 (1–2 спринта)**
- **P2 (регулярный процесс)**

Для каждого пункта: владелец (Backend/DevOps/Sec), ETA, критерий завершения.

## 4) Важные требования к качеству

- Не ограничивайся общими словами — только проверяемые факты.
- Любой вывод подтверждай ссылкой на файл/строки.
- Если в чем-то не уверен — пометь как Hypothesis и предложи как проверить.
- Покажи быстрые win fixes (которые можно сделать за 1 день).
- Пометь отдельно всё, что может привести к утечке данных или обходу авторизации.

## 5) Дополнительно (опционально, но желательно)

- Сформируй короткий security checklist для PR review команды.
- Сформируй шаблон policy для ротации ключей (90 дней).
- Сформируй минимальный baseline для CI security gates:
  - secret scan
  - dependency audit
  - SAST
  - DAST (staging)
