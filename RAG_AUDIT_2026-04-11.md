# Комплексный аудит RAG-пайплайна (2026-04-11)

## 1) Цепочка от запроса пользователя до ответа модели

1. **Вход запроса**: `POST /api/chat`, проверка invite-кода, авторизации диалога, получение `messages`, `conversationId`, вложений.  
2. **Подготовка контекста**: объединение текущих и session-документов, определение `docIntent`, расширение поискового запроса именами файлов и фрагментами вложений.  
3. **Параллельные классификаторы**: intent-классификация, off-topic-классификация, загрузка/сохранение памяти диалога.  
4. **Маршрутизация retrieval**: выбор deterministic vs agentic RAG через `isComplexQuery(...)`.  
5. **Детерминированный retrieval**: hybrid search + section/document/catalog lookups + доп. intent-aware поиски + regime post-filter.  
6. **Agentic retrieval**: инструментальные вызовы `search_knowledge_base`, `lookup_section`, `lookup_document` через Gemini tool-calling, накопление чанков.  
7. **Реранк и фильтрация**: `intentAwareRerank` → `rerank` (Gemini/Voyage) → `filterByRelevance`.  
8. **Постобработка**: enrichment оригиналами для денормализованных источников, relationship-expansion, загрузка изображений чанков.  
9. **Сборка системного промпта**: большой policy-блок + RAG XML-контекст + правила по режимам/НДС/директивам + uploaded-docs.  
10. **Вызов модели**: `@google/genai` stream (`gemini-3-flash-preview`), `systemInstruction=systemPrompt`, стрим ответа клиенту.

## 2) Найденные риски и узкие места

### 2.1 Реранкер пропускал шум
- Веса были смещены в сторону исходной hybrid similarity (60/40), поэтому тематически похожий, но нецелевой мусор иногда переживал rerank.
- Не было «жёсткого подавления» для фрагментов с низким LLM score и почти нулевым лексическим пересечением с запросом.

### 2.2 Маршрутизация в agentic RAG была бинарной и грубой
- `isComplexQuery` был правилом «если совпал один паттерн — сразу agentic».  
- Это может увеличивать долю agentic-запросов там, где deterministic даёт стабильнее/дешевле/чище.

### 2.3 Системный промпт очень перегружен
- В одном системном сообщении одновременно: anti-injection, формат, таблицы, режимы, НДС, курсы, SPU-реестр и др.
- Риск: рост длины и конкуренция инструкций; при длинном контексте часть поведения может проявляться нестабильно.

## 3) Выполненные улучшения в коде

### 3.1 Усилен LLM rerank и отсечение мусора
- Повышен вклад LLM cross-encoder оценки (с 40% до 65%), снижен вклад исходной similarity (до 35%).
- Добавлено жёсткое подавление кандидатов: при низком LLM score и минимальном token overlap итоговый балл режется на 75%.
- Добавлен небольшой бонус для действительно сильных фрагментов (LLM score ≥ 7.5) для большего отрыва релевантного топа.
- Ужесточён текст промпта реранкера (явно занижать score для «похожих, но не отвечающих» и для не того режима/организации).

### 3.2 Улучшено ранжирование запросов на agentic RAG
- `isComplexQuery` переведён из жёсткого yes/no на score-based маршрутизацию.
- Учитываются несколько сигналов сложности с весами (both-regimes, comparison keywords, multi-section и др.).
- Добавлен штраф за низкую уверенность intent-классификатора (чтобы реже зря уходить в agentic).
- Добавлено логирование причины маршрутизации (`complexity score`, `reasons`) для дальнейшего анализа.

## 4) Проверка системных промптов (что реально уходит в модель)

### Основной ответный промпт
- Формируется в `systemPrompt` внутри `chat.ts`, включает:
  - строгий anti-prompt-injection policy;
  - режимные правила 223/вне 223;
  - формат, таблицы, полноту ответа;
  - динамические блоки: low-confidence warning, directives, uploaded-doc instructions, multimodal hints;
  - XML-контекст `<documents>` + `<uploaded_documents>`.

### Agentic промпт
- Формируется отдельной строкой `agenticPrompt` и отправляется как user-content в agent loop.
- Содержит явные требования вызывать tools по каждой сущности/режиму и завершать после достаточного покрытия.

## 5) Рекомендации следующего шага

1. **A/B-оценка качества retrieval** на offline-наборе (precision@k, nDCG@10, доля off-topic в top-5).  
2. **Разделить system prompt на компактные policy-блоки** (например, обязательные ядро-правила + доменные add-ons по intent), чтобы снизить конфликтность длинных инструкций.  
3. **Добавить телеметрию**: какая доля agentic-переходов приводит к улучшению answer quality относительно deterministic.
4. **Ввести whitelist/blacklist источников по интентам** (например, SPU-карточки не пускать в общие нормативные ответы без явного запроса).
