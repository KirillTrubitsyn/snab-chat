# Повторный аудит RAG после последних изменений (2026-04-11)

## Контекст проверки
Проведён повторный аудит всего пайплайна: от входа пользовательского сообщения до вызова модели, включая:
- маршрутизацию deterministic/agentic;
- retriever/hybrid search + вспомогательные lookup;
- реранкер (Gemini/Voyage), фильтрацию релевантности;
- системные промпты, которые фактически уходят в модель.

---

## 1) Сквозной путь запроса (user → model)

1. `POST /api/chat` принимает `messages`, `conversationId`, вложенные документы.  
2. Запрос обогащается контекстом (вложения, follow-up контекст, сущности из истории).  
3. Параллельно запускаются:
   - `classifyIntent(...)`,
   - `classifyOffTopic(...)`,
   - загрузка/сохранение памяти диалога.
4. Роутинг в retrieval:
   - `isComplexQuery(...)` => **agentic** или **deterministic** путь.
5. Retrieval:
   - deterministic: `hybridSearch` + section/doc/catalog lookup + intent supplementary;
   - agentic: tools (`search_knowledge_base`, `lookup_section`, `lookup_document`) через tool-calling.
6. Релевантность:
   - `intentAwareRerank(...)` (tag/tier boosts),
   - `rerank(...)` (Gemini/Voyage),
   - `filterByRelevance(...)`.
7. Сборка итогового контекста:
   - XML `<documents>` (+ `<uploaded_documents>`),
   - инструкции по режимам, таблицам, цитированию, anti-injection и пр.
8. Вызов Gemini stream через `@google/genai` с `systemInstruction: systemPrompt`.

---

## 2) Что проверено по реранкеру

## 2.1 Gemini reranker
**Текущее состояние:**
- Ужесточён prompt реранкера: прямо требует занижать оценку для тематически похожего, но не отвечающего контента.
- В blend вес LLM повышен (0.65), исходного similarity снижен (0.35).
- Добавлен lexical overlap и hard suppression для низкого LLM score + низкого overlap.

**Вывод:**
- Это действительно снижает шанс протаскивания «семантического мусора» в top.
- По логике — улучшение корректное и направленное на precision.

**Зона риска, которую ещё стоит закрыть:**
- `filterByRelevance` по-прежнему использует фиксированный абсолютный порог (`0.35`) без адаптации к типу интента/корпусу; это может быть слишком жёстко/мягко в разных классах вопросов.

## 2.2 Voyage reranker
**Текущее состояние:**
- Если выбран `RERANKER_MODEL=voyage`, используется отдельный pipeline с blend 50/50.

**Вывод:**
- Конфигурация корректна, но логика suppression/overlap есть только в Gemini-ветке. Для паритета качества стоит унифицировать post-processing независимо от модели реранкера.

## 3) Agentic RAG: как ранжируются вопросы на этот путь

**Текущее состояние:**
- `isComplexQuery(...)` теперь score-based (а не одномоментный boolean по одному паттерну).
- Учитываются несколько сигналов сложности и штраф за низкую уверенность intent.
- Добавлено логирование причин маршрутизации.

**Вывод:**
- Роутинг стал более контролируемым и прозрачным.
- Ожидаемо уменьшит лишние уходы в agentic при простых/шумных запросах.

**Оставшийся риск:**
- Порог (`score >= 3`) и веса сигналов пока статические; без оффлайн-метрик возможен овер/андер-роутинг на части доменов.

## 4) Проверка системных промптов, уходящих в модель

## 4.1 Основной промпт ответа
- Формируется в `systemPrompt` перед генерацией ответа.
- Включает:
  - anti-prompt-injection,
  - строгие ограничения source-grounding,
  - формат ответа/таблиц,
  - доменные блоки (223/вне 223, SPU и т.д.),
  - динамические предупреждения/директивы,
  - RAG XML-контекст.

**Наблюдение:**
- Промпт очень большой (порядка ~14.8k символов только до мультимодальной сборки), что повышает риск конкуренции инструкций.

## 4.2 Agentic промпт
- Формируется отдельно и отправляется в tool-calling цикл.
- Содержит хорошие guardrails: покрытие всех режимов/организаций, обязательные вызовы lookup при явных ссылках.

**Наблюдение:**
- По структуре корректен; логика pre-seed по сущностям дополнительно страхует от однобокого поиска.

## 5) Приоритетные рекомендации (после повторного аудита)

### P0 (рекомендую внедрить первым)
1. Добавить **eval-набор** (50–100 реальных запросов) и считать:
   - precision@5,
   - долю off-topic в top-5,
   - hit-rate на «правильный режим» (223/non-223).
2. Для `isComplexQuery` ввести конфигурируемые веса/порог (через env), чтобы не деплоить код на каждую подстройку.

### P1
3. Разделить длинный `systemPrompt` на 2 уровня:
   - неизменяемое «ядро политики»;
   - доменные add-on блоки по интенту (подключать только при необходимости).
4. Унифицировать post-processing после реранка (Gemini/Voyage), чтобы suppression логика не зависела от выбранной модели.

### P2
5. Ввести lightweight telemetry по agentic:
   - сколько tool-вызовов,
   - прирост качества относительно deterministic fallback,
   - средняя стоимость и latency по классу запроса.
