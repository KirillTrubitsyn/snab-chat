# СнабЧат — RAG-ассистент Дирекции по закупкам

## Обзор проекта

Next.js 15 приложение — чат-бот на базе RAG (Retrieval-Augmented Generation), отвечающий строго по загруженным документам (DOCX, PDF, XLSX). Использует Supabase (PostgreSQL + pgvector) для хранения и поиска, Google Gemini для генерации и эмбеддингов. Поддерживает голосовой ввод и фотографирование с камеры телефона.

## Технологический стек

- **Фреймворк:** Next.js 15 (App Router)
- **LLM:** Google Gemini 3 Flash (`gemini-3-flash`) через `@ai-sdk/google` + Vercel AI SDK
- **Эмбеддинги:** `gemini-embedding-2-preview` (1536 dim) через `@google/genai`
- **БД:** Supabase PostgreSQL + pgvector
- **Парсинг:** mammoth (DOCX), pdf-parse (PDF), xlsx (Excel)
- **Стриминг:** Vercel AI SDK `streamText` + data stream protocol

## Архитектура

```
Пользователь → Chat.tsx (React) → /api/chat (Next.js API Route)
                                      ├── hybridSearch() → Supabase RPC `hybrid_search`
                                      ├── filterByRelevance() → порог 0.35, макс 8 чанков
                                      ├── loadConversationContext() → история + резюме
                                      └── streamText(gemini-3-flash) → стриминг ответа
```

## Голосовой ввод (мобильные устройства и ПК)

Пользователь может надиктовать сообщение голосом вместо набора текста. Реализовано через Web Speech API (`SpeechRecognition`).

**Компонент:** `components/chat/VoiceButton.tsx`

**Как работает:**
1. Пользователь нажимает кнопку микрофона в области ввода
2. Браузер запрашивает разрешение на доступ к микрофону (при первом использовании)
3. Распознавание речи работает в режиме реального времени (язык: `ru-RU`)
4. Включён непрерывный режим (`continuous: true`) и промежуточные результаты (`interimResults: true`)
5. При остановке записи финальный текст добавляется в поле ввода (дописывается к существующему тексту)
6. Пользователь может отредактировать текст и отправить сообщение

**Особенности:**
- Поддерживает `SpeechRecognition` и `webkitSpeechRecognition` (Safari)
- Автоматический перезапуск при обрыве соединения
- Визуальная индикация записи (кнопка подсвечивается)
- Кнопка заблокирована при отправке сообщения (`disabled`)

**Ограничения:**
- Web Speech API требует подключения к интернету (распознавание выполняется на серверах браузера)
- Поддержка зависит от браузера (Chrome, Safari, Edge — поддерживают; Firefox — частично)

## Фотографирование с камеры телефона

Пользователь может сфотографировать документ или объект камерой телефона и загрузить фото прямо в чат. Текст с фото извлекается автоматически через Gemini Vision (OCR).

**Компонент:** `components/chat/CameraButton.tsx`

**Как работает:**
1. Пользователь нажимает кнопку камеры в области ввода
2. Открывается полноэкранный модал с видеопотоком камеры (`getUserMedia`)
3. Можно переключаться между фронтальной и задней камерой
4. Нажатие кнопки затвора захватывает кадр (JPEG, качество 85%, разрешение до 1920×1080)
5. Фото отправляется на `/api/parse`, где Gemini Vision извлекает текст (OCR)
6. Извлечённый текст (markdown) прикрепляется к сообщению как документ
7. При отправке сообщения текст из фото передаётся в RAG-контекст модели

**Особенности:**
- Лимит: до 10 фото за одно сообщение (`MAX_CHAT_PHOTOS = 10`)
- Превью загруженных фото отображается в панели над полем ввода
- Индикация статуса: обработка / готово / ошибка (бейджи на превью)
- Можно удалить отдельное фото до отправки
- Также можно загрузить фото из галереи (через стандартный файловый ввод — JPG, PNG, GIF, BMP, WEBP)

**Поток обработки фото:**
```
Камера → JPEG → /api/parse → parseImageToMarkdown() → Gemini Vision OCR → markdown
                                                                              ↓
                                                              attachedDocuments → /api/chat → RAG-контекст
```

**Ограничения:**
- Требуется разрешение на доступ к камере
- Максимальный размер файла: 4 МБ (Vercel body limit), файлы больше 4 МБ загружаются через Supabase Storage
- Качество OCR зависит от чёткости фото и освещения

## Принципы ответов модели

Системный промпт настроен так, чтобы модель:
1. Отвечала ТОЛЬКО по предоставленным документам (XML-формат `<documents>`)
2. Цитировала источники в формате `[doc:N]` после каждого утверждения
3. Явно указывала, когда информация отсутствует в базе знаний
4. Не использовала общие знания для заполнения пробелов
5. При низкой релевантности (`lowConfidence`) предупреждала пользователя

## Важные моменты при доработке

- **НЕ менять** формат системного промпта без тестирования — он тщательно настроен против галлюцинаций
- Фильтрация релевантности (`filterByRelevance`) — критический компонент, пороги подбирались эмпирически
- Контекст передаётся модели в XML-формате `<documents>` — это помогает модели различать источники
- Источники в UI показываются только из релевантных отфильтрованных чанков (не из всех 20)
- Стриминг реализован вручную (два пути: новый диалог + существующий), а не через стандартный `useChat`
- Голосовой ввод (`VoiceButton`) использует Web Speech API браузера — распознавание происходит на стороне браузера, а не через наш бэкенд
- Загрузка аудиофайлов (MP3/WAV) — отдельный путь: файл отправляется на `/api/parse`, где транскрибируется через Gemini (`transcribeAudioWithGemini`)
- Фото с камеры (`CameraButton`) обрабатываются через Gemini Vision OCR на сервере — извлечённый текст используется как документ в RAG-контексте
